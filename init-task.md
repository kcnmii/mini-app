# ТЗ: Telegram Mini App для счетов и простых КП

## 1. Идея проекта

Нужно создать **Telegram Mini App** для малого и среднего бизнеса, который позволяет:

* быстро создавать **счета**
* быстро создавать **простые коммерческие предложения**
* использовать **встроенные шаблоны**
* в будущем загружать **свои простые шаблоны**
* выбирать клиента, контакт и позиции из каталога
* генерировать **PDF**
* при необходимости формировать **DOCX**
* сохранять историю документов
* делиться документом через Telegram / email / любой канал через системную кнопку “Поделиться”

Продукт должен быть максимально простым, быстрым и понятным. Основной сценарий:

> пользователь открывает Mini App из Telegram, создаёт документ за 1–3 минуты и сразу отправляет его клиенту.

---

## 2. Главная продуктовая логика

Это **не CRM**, **не ERP**, **не склад**, **не бухгалтерия в полном виде**.

Это:

* быстрый генератор документов
* поверх:

  * клиентов
  * контактов
  * каталога товаров/услуг
  * шаблонов документов

### Базовый сценарий MVP

1. Пользователь открывает Mini App
2. Авторизуется через Telegram автоматически
3. Выбирает:

   * клиент
   * контакт
   * тип документа
4. Добавляет позиции
5. Заполняет базовые поля документа
6. Нажимает “Сгенерировать”
7. Получает PDF
8. Нажимает “Поделиться”

## 4. Общая архитектура

## 4.1 Telegram Mini App модель

Telegram Mini App — это **web app внутри Telegram**, который запускается из бота, профиля бота или по специальной ссылке. Telegram передаёт приложению launch parameters и `init data`, которые должны использоваться для авторизации через backend. В production для Mini App нужен **HTTPS**.

### Состав системы:

1. **Telegram Bot**
2. **Mini App frontend**
3. **Backend API**
4. **База данных**
5. **Storage**
6. **PDF render service**

---

## 4.2 Рекомендуемый стек

### Frontend

* React
* Telegram Mini Apps SDK для React: `@tma.js/sdk-react`
* роутинг внутри Mini App
* адаптивный mobile-first UI

Документация Mini Apps рекомендует использовать SDK-пакет, соответствующий целевой библиотеке, а для React — `@tma.js/sdk-react`. Также есть генератор стартового проекта `@tma.js/create-mini-app`.

### Backend

* FastAPI
* PostgreSQL
* Redis (очереди, кэш, фоновые задачи)
* Celery / RQ / dramatiq (по выбору)
* SQLAlchemy
* Pydantic

### PDF

* Gotenberg в Docker-контейнере
  Gotenberg официально разворачивается как отдельный сервис в Docker и используется для генерации PDF из HTML / офисных документов.

### Bot

* Telegram Bot API
* кнопка открытия Mini App
* отправка файлов в чат через `sendDocument`

Telegram Bot API позволяет ботам отправлять документы, а Mini Apps официально интегрируются в экосистему ботов.

### Storage

* S3-compatible storage
* на старте допустимо:

  * MinIO на отдельном сервере
  * или Backblaze B2 / Cloudflare R2
* хранить:

  * PDF
  * DOCX
  * изображения
  * логотипы
  * аватары
  * шаблоны

---

## 4.3 Архитектура серверов на старте

### Вариант старта (рекомендуется)

**Сервер 1: основной**

* FastAPI
* PostgreSQL
* Redis
* frontend
* bot backend

**Сервер 2: PDF render**

* Gotenberg

**Сервер 3 (опционально): Storage**

* MinIO
  или внешний S3-compatible storage

### Почему так

* рендер PDF не мешает API
* документы и файлы вынесены из основного диска
* можно масштабировать отдельно backend и render

## 5. Telegram логика и авторизация

## 5.1 Как пользователь входит

1. Пользователь открывает бота
2. Нажимает кнопку “Открыть приложение”
3. Telegram открывает Mini App
4. Mini App получает launch params и `tgWebAppData`
5. Frontend отправляет init data на backend
6. Backend валидирует подпись Telegram и создаёт/обновляет локального пользователя

Telegram Mini Apps docs прямо указывают, что init data может использоваться как фактор авторизации, но её нужно валидировать на сервере.

## 5.2 Что обязательно использовать в Mini App

* Main Button
* Back Button
* Popup
* Theme / viewport integration

Эти элементы входят в стандартный набор методов платформы Mini Apps.

## 8. Логика AI

### Главное правило

AI не является обязательной частью всех документов.

## Где AI используется

1. Простые КП

   * генерация текста блоков
2. Простые пользовательские шаблоны

   * подсказка, где должны быть ключи
   * автозаполнение некоторых полей
3. Позже:

   * предложения по структуре
   * улучшение текста

## Где AI НЕ обязателен

* Счета
* Выбор клиента
* Выбор товаров/услуг
* Генерация PDF
* Поделиться

Это принципиально важно для экономики и стабильности.

## 10. Генерация PDF и DOCX

## 10.1 PDF

HTML шаблон собирается на backend, затем отправляется в Gotenberg.

### Поток:

1. backend собирает `content_json`
2. рендерит HTML
3. отправляет в Gotenberg
4. получает PDF
5. сохраняет PDF в storage
6. отдаёт ссылку и метаданные

Gotenberg поддерживает конвертацию и работает как отдельный HTTP-сервис в Docker.

## 10.2 DOCX

На старте можно:

* выдавать DOCX только по системным шаблонам
  или
* отложить DOCX до второго этапа

Если делать DOCX:

* использовать шаблонизатор
* не обещать сложную верстку
* держать простые шаблоны

## 15. Definition of Done MVP

MVP считается готовым, когда пользователь может:

1. Открыть Mini App из Telegram
2. Авторизоваться автоматически
3. Создать клиента и контакт
4. Создать позицию в каталоге
5. Создать счёт
6. Получить PDF
7. Нажать “Поделиться”
8. Открыть историю документа позже

Вторая ступень MVP:
9. Создать простое КП
10. Получить PDF / DOCX
11. Поделиться

---

## 16. Что передать разработчику как главный принцип

> Мы строим Telegram Mini App как быстрый генератор документов.
> Основной сценарий — счёт за 1–3 минуты.
> Вся архитектура должна быть простой: bot → mini app frontend → backend → PDF render → storage.
> AI не должен быть обязательной частью продукта.
> Документы должны генерироваться и без AI.
> Всё, что усложняет путь до первого документа, не относится к MVP.
